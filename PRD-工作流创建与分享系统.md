## 1. 背景与目标

### 1.1 业务痛点
- 用户需要记录和整理工作流程，但现有工具过于复杂（如需审批、组织架构管理）
- 缺乏轻量级的工作流记录工具，仅用于备忘和分享
- 工作流分享不够便捷，无法快速生成只读页面供他人查看
- 现有工具不支持版本历史，误操作后难以恢复
- 现有工具节点类型单一，无法满足多样化场景

### 1.2 期望价值
- 提供简洁的工作流创建体验，降低使用门槛
- 支持灵活的工作流分享，便于知识传递和协作
- 保留历史版本，确保工作流变更可追溯
- 支持导出为标准格式（Markdown），便于存档和迁移
- **支持多种节点类型，适应不同业务场景**
- **支持逻辑分支，表达复杂的条件流程**

### 1.3 成功指标
| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 工作流创建完成率 | ≥ 95% | 创建成功数/创建开始数 |
| 页面加载时间 | ≤ 2秒 | 前端性能监控 |
| 分享链接访问成功率 | ≥ 99% | 分享链接有效访问数/总访问数 |
| 用户满意度 | ≥ 4.0/5.0 | 用户调研问卷 |
| 功能使用率 | ≥ 80% | 使用分享功能的用户数/总用户数 |
| 节点类型使用率 | ≥ 3种/用户 | 每个用户平均使用的节点类型数 |

## 3. 工作流节点类型详解（核心章节）

### 3.1 节点类型概述

本系统支持**6种核心节点类型**，每种节点类型都有特定的用途、字段结构和展示方式。用户可以根据实际业务场景选择合适的节点类型来构建工作流。

| 节点类型 | 类型标识 | 适用场景 | 复杂度 |
|----------|----------|----------|--------|
| **标准节点** | standard | 通用流程步骤 | ⭐ |
| **判断节点** | decision | 条件分支、决策点 | ⭐⭐⭐ |
| **并行节点** | parallel | 同时执行的多个任务 | ⭐⭐ |
| **循环节点** | loop | 重复执行的任务 | ⭐⭐ |
| **子流程节点** | subflow | 嵌套子工作流 | ⭐⭐⭐ |
| **备注节点** | note | 说明、注释信息 | ⭐ |

### 3.2 标准节点（Standard Node）

#### 3.2.1 节点定义
标准节点是最基础的节点类型，用于描述一个具体的流程步骤，包含人物、时间、地点、事件等完整信息。

#### 3.2.2 数据结构

```json
{
  "id": "uuid",
  "type": "standard",
  "title": "节点标题",
  "position": 1,
  "data": {
    "characters": ["张三", "李四"],
    "scheduledAt": "2026-05-10T14:00:00Z",
    "location": "会议室A",
    "event": {
      "content": "事件详细描述",
      "format": "rich-text"
    },
    "attachments": [
      {
        "name": "会议资料.pdf",
        "url": "https://...",
        "type": "application/pdf"
      }
    ],
    "tags": ["重要", "评审"],
    "status": "pending"
  },
  "config": {
    "showInTimeline": true,
    "color": "#1890ff",
    "icon": "check-circle"
  }
}
```

#### 3.2.3 字段说明

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| **基础信息** |
| id | UUID | 是 | 自动生成 | 节点唯一标识 |
| type | String | 是 | "standard" | 节点类型 |
| title | String | 是 | - | 节点标题，≤100字符 |
| position | Number | 是 | 自动递增 | 节点顺序 |
| **数据字段** |
| data.characters | Array | 否 | [] | 参与人物列表 |
| data.scheduledAt | DateTime | 否 | null | 计划执行时间 |
| data.location | String | 否 | "" | 地点，≤255字符 |
| data.event.content | String | 否 | "" | 事件描述，富文本 |
| data.event.format | Enum | 是 | "rich-text" | 格式：rich-text/markdown/plain |
| data.attachments | Array | 否 | [] | 附件列表 |
| data.tags | Array | 否 | [] | 标签列表 |
| data.status | Enum | 否 | "pending" | 状态：pending/in-progress/completed/skipped |
| **配置字段** |
| config.showInTimeline | Boolean | 否 | true | 是否显示在时间线 |
| config.color | String | 否 | "#1890ff" | 节点颜色 |
| config.icon | String | 否 | "check-circle" | 节点图标 |

#### 3.2.4 使用场景

| 场景 | 示例 |
|------|------|
| 会议安排 | 需求评审会议，指定参与人、时间、会议室 |
| 任务分配 | 开发任务分配给具体人员，设定截止时间 |
| 里程碑标记 | 项目关键节点，如"完成设计稿" |
| 审批环节 | 需要某人审批的步骤 |

### 3.3 判断节点（Decision Node）

#### 3.3.1 节点定义
判断节点用于表示流程中的决策点，根据条件判断结果将流程导向不同的分支路径。支持二元判断（是/否）和多元判断（多选项）。

#### 3.3.2 数据结构

```json
{
  "id": "uuid",
  "type": "decision",
  "title": "技术方案评审结果",
  "position": 2,
  "data": {
    "condition": "技术方案是否通过评审？",
    "decisionType": "binary",
    "branches": [
      {
        "id": "branch-yes",
        "label": "通过",
        "condition": "评审得分 ≥ 80分",
        "color": "#52c41a",
        "nextNodeId": "uuid-standard-3",
        "description": "进入开发阶段"
      },
      {
        "id": "branch-no",
        "label": "不通过",
        "condition": "评审得分 < 80分",
        "color": "#f5222d",
        "nextNodeId": "uuid-standard-4",
        "description": "返回修改方案"
      }
    ],
    "defaultBranch": "branch-no"
  },
  "config": {
    "showCondition": true,
    "showBranchLabels": true,
    "layout": "horizontal"
  }
}
```

#### 3.3.3 字段说明

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| **基础信息** |
| id | UUID | 是 | 自动生成 | 节点唯一标识 |
| type | String | 是 | "decision" | 节点类型 |
| title | String | 是 | - | 节点标题 |
| position | Number | 是 | 自动递增 | 节点顺序 |
| **数据字段** |
| data.condition | String | 是 | - | 判断条件描述 |
| data.decisionType | Enum | 是 | "binary" | 判断类型：binary(二元)/multiple(多元) |
| data.branches | Array | 是 | [] | 分支列表 |
| data.branches[].id | String | 是 | - | 分支唯一标识 |
| data.branches[].label | String | 是 | - | 分支标签 |
| data.branches[].condition | String | 否 | - | 分支条件描述 |
| data.branches[].color | String | 否 | 自动分配 | 分支颜色 |
| data.branches[].nextNodeId | UUID | 否 | null | 下一节点ID |
| data.branches[].description | String | 否 | - | 分支说明 |
| data.defaultBranch | String | 否 | null | 默认分支ID |
| **配置字段** |
| config.showCondition | Boolean | 否 | true | 是否显示条件 |
| config.showBranchLabels | Boolean | 否 | true | 是否显示分支标签 |
| config.layout | Enum | 否 | "horizontal" | 布局：horizontal/vertical |

#### 3.3.4 判断类型详解

**二元判断（Binary Decision）**
- 适用场景：简单的Yes/No决策
- 分支数量：固定2个
- 默认标签：通过/不通过、是/否、成功/失败

**多元判断（Multiple Decision）**
- 适用场景：多选项决策
- 分支数量：2-6个（建议不超过4个以保证可读性）
- 示例：优先级（高/中/低）、状态（待处理/处理中/已完成/已取消）

#### 3.3.5 使用场景

| 场景 | 判断类型 | 分支示例 |
|------|----------|----------|
| 需求评审 | 二元 | 通过 → 进入开发；不通过 → 返回修改 |
| 技术方案选择 | 多元 | 方案A / 方案B / 方案C |
| 风险评估 | 多元 | 高风险 / 中风险 / 低风险 |
| 审批流程 | 二元 | 同意 → 继续；拒绝 → 结束 |
| 用户类型判断 | 多元 | 新用户 / 老用户 / VIP用户 |

### 3.4 并行节点（Parallel Node）

#### 3.4.1 节点定义
并行节点用于表示可以同时执行的多个任务或子流程。这些任务之间没有依赖关系，可以同时进行。

#### 3.4.2 数据结构

```json
{
  "id": "uuid",
  "type": "parallel",
  "title": "开发阶段并行任务",
  "position": 3,
  "data": {
    "description": "前后端开发可以并行进行",
    "tasks": [
      {
        "id": "task-1",
        "title": "前端开发",
        "assignee": "前端工程师A",
        "status": "in-progress",
        "priority": "high",
        "estimatedHours": 40,
        "dependencies": []
      },
      {
        "id": "task-2",
        "title": "后端开发",
        "assignee": "后端工程师B",
        "status": "in-progress",
        "priority": "high",
        "estimatedHours": 48,
        "dependencies": []
      },
      {
        "id": "task-3",
        "title": "数据库设计",
        "assignee": "DBA",
        "status": "completed",
        "priority": "medium",
        "estimatedHours": 16,
        "dependencies": []
      }
    ],
    "completionCondition": "all",
    "syncPoint": {
      "enabled": true,
      "description": "等待所有开发任务完成后进入联调"
    }
  },
  "config": {
    "showProgress": true,
    "showAssignee": true,
    "layout": "grid"
  }
}
```

#### 3.4.3 字段说明

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| **基础信息** |
| id | UUID | 是 | 自动生成 | 节点唯一标识 |
| type | String | 是 | "parallel" | 节点类型 |
| title | String | 是 | - | 节点标题 |
| position | Number | 是 | 自动递增 | 节点顺序 |
| **数据字段** |
| data.description | String | 否 | - | 并行任务说明 |
| data.tasks | Array | 是 | [] | 并行任务列表 |
| data.tasks[].id | String | 是 | - | 任务唯一标识 |
| data.tasks[].title | String | 是 | - | 任务标题 |
| data.tasks[].assignee | String | 否 | - | 负责人 |
| data.tasks[].status | Enum | 否 | "pending" | 状态 |
| data.tasks[].priority | Enum | 否 | "medium" | 优先级 |
| data.tasks[].estimatedHours | Number | 否 | 0 | 预估工时 |
| data.tasks[].dependencies | Array | 否 | [] | 依赖任务ID列表 |
| data.completionCondition | Enum | 否 | "all" | 完成条件：all/any/majority/n |
| data.syncPoint.enabled | Boolean | 否 | true | 是否启用同步点 |
| data.syncPoint.description | String | 否 | - | 同步点说明 |
| **配置字段** |
| config.showProgress | Boolean | 否 | true | 是否显示进度 |
| config.showAssignee | Boolean | 否 | true | 是否显示负责人 |
| config.layout | Enum | 否 | "grid" | 布局：grid/list |

#### 3.4.4 完成条件类型

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| **all** | 所有任务完成 | 所有子任务都必须完成才能继续 |
| **any** | 任一任务完成 | 多个方案并行，有一个完成即可 |
| **majority** | 多数任务完成 | 超过50%的任务完成即可 |
| **n** | 指定数量完成 | 至少n个任务完成 |

#### 3.4.5 使用场景

| 场景 | 示例 |
|------|------|
| 并行开发 | 前端、后端、数据库设计同时进行 |
| 多方案并行 | 同时探索多个技术方案，选择最优 |
| 多人协作 | 不同角色同时处理各自任务 |
| 并行审批 | 多个部门同时审批，无需等待 |

### 3.5 循环节点（Loop Node）

#### 3.5.1 节点定义
循环节点用于表示需要重复执行的任务或流程，直到满足特定条件才退出循环。支持固定次数循环和条件循环两种模式。

#### 3.5.2 数据结构

```json
{
  "id": "uuid",

    "loopType": "conditional",
    "maxIterations": 5,
    "exitCondition": "审查通过",

      {
        "iteration": 1,
        "status": "completed",
        "result": "未通过",
        "feedback": "存在内存泄漏问题",

      {
        "iteration": 2,
        "status": "completed",
        "result": "未通过",
        "feedback": "代码规范不符合要求",
        "completedAt": "2026-05-16T14:00:00Z",
        "assignee": "开发工程师A"
      },
      {
        "iteration": 3,
        "status": "in-progress",
        "result": null,
        "feedback": null,
        "completedAt": null,
        "assignee": "开发工程师A"
      }
    ],
    "currentIteration": 3,
    "bodyNodeId": "uuid-standard-review"

    "showIterationHistory": true,
    "showProgress": true,
    "allowSkip": false
```

#### 3.5.3 字段说明

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| **基础信息** |
| id | UUID | 是 | 自动生成 | 节点唯一标识 |
| type | String | 是 | "loop" | 节点类型 |

#### 3.5.4 循环类型详解

**固定次数循环（Fixed Loop）**
- 适用场景：明确知道需要执行多少次
- 必须设置：循环次数（1-100）
- 示例：每日站会（5个工作日）、代码审查（最多3轮）

**条件循环（Conditional Loop）**
- 适用场景：直到满足某个条件才停止
- 必须设置：退出条件描述、最大迭代次数（防止死循环）
- 示例：直到测试通过、直到客户确认

#### 3.5.5 使用场景

| 场景 | 循环类型 | 示例 |
|------|----------|------|
| 代码审查 | 条件循环 | 直到审查通过，最多3次 |
| 迭代开发 | 固定次数 | Sprint周期，固定2周 |
| 测试修复 | 条件循环 | 直到所有Bug修复 |
| 每日站会 | 固定次数 | 每个Sprint的5个工作日 |
| 方案优化 | 条件循环 | 直到满足性能指标 |

#### 3.6.1 节点定义
子流程节点用于引用另一个工作流作为当前流程的一部分，实现流程的模块化和复用。支持嵌入式和引用式两种子流程类型。

```json
      }

        "source": "output.testResult",

        "description": "测试结果"

      {
```

| data.subflowId | UUID | 是（引用式） | null | 引用的工作流ID |

| config.showProgress | Boolean | 否 | true | 是否显示进度 |
| config.expandable | Boolean | 否 | true | 是否可展开查看详情 |

#### 3.6.4 子流程类型详解

**嵌入式子流程（Embedded Subflow）**
- 适用场景：子流程仅在此工作流中使用，不需要独立存在
- 特点：子流程节点内部直接定义流程步骤
- 优势：封装性好，不污染工作流列表
- 限制：不能在其他工作流中复用

**引用式子流程（Reference Subflow）**
- 适用场景：多个工作流共享相同的子流程
- 特点：引用已存在的工作流
- 优势：一处修改，多处生效；支持版本控制
- 限制：被引用工作流删除会影响引用者

#### 3.6.5 执行模式详解

**同步执行（Synchronous）**
- 父流程等待子流程完成后才继续
- 适用：子流程是必经步骤

**异步执行（Asynchronous）**
- 父流程不等待，子流程在后台执行
- 适用：子流程是辅助流程，如日志记录、通知发送

#### 3.6.6 使用场景

| 场景 | 子流程类型 | 示例 |
|------|----------|------|
| 标准审批流程 | 引用式 | 多个项目共用相同的审批流程 |
| 测试流程 | 引用式 | 所有功能上线前执行标准测试 |
| 文档审核 | 嵌入式 | 仅在此流程中使用的特殊审核 |
| 数据备份 | 引用式 | 所有操作前自动执行备份 |
| 通知流程 | 异步引用 | 发送邮件/消息，不阻塞主流程 |

## 4. 节点连接与逻辑分支

### 4.1 连接类型

节点之间的连接表示流程的执行顺序和逻辑关系。系统支持以下连接类型：

| 连接类型 | 说明 | 视觉样式 |
|----------|------|----------|
| **顺序连接** | 默认的流程执行顺序 | 实线箭头 → |
| **条件连接** | 判断节点的分支连接 | 带标签的实线箭头 |
| **循环连接** | 返回循环节点的连接 | 虚线箭头 ⟲ |
| **并行连接** | 并行节点的任务连接 | 双实线箭头 ⇉ |
| **子流程连接** | 子流程的输入/输出 | 点线箭头 ⇢ |

### 4.2 连接数据结构

```json
{
  "id": "uuid",
  "source": "uuid-node-a",
  "target": "uuid-node-b",
  "type": "conditional",
  "label": "通过",
  "condition": "评审得分 ≥ 80",
  "style": {
    "color": "#52c41a",
    "lineStyle": "solid",
    "arrowType": "arrow"
  },
  "metadata": {
    "createdAt": "2026-05-01T10:00:00Z",
    "description": "评审通过后的流程走向"
  }
}
```

### 4.3 连接规则

| 规则 | 说明 | 示例 |
|------|------|------|
| **单入多出** | 标准节点可以有多个出连接（分支） | 标准节点 → 判断节点 |
| **多入单出** | 多个节点可以连接到同一个目标 | 两个分支合并到同一节点 |
| **禁止循环** | 不允许形成无限循环（除非使用循环节点） | A→B→A 被禁止 |
| **判断必连** | 判断节点的每个分支必须有下一节点 | 每个分支都需指定nextNodeId |
| **并行同步** | 并行节点的所有任务完成后才能继续 | 同步点机制 |

### 4.4 逻辑分支示例

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           复杂逻辑分支示例                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
```

## 5. 用户故事

### 5.1 工作流管理相关

#### US-001：创建工作流
**作为** 一位项目经理，
**我希望** 能够创建一个新的工作流，
**以便** 记录项目的关键步骤和流程。

**验收标准：**
- 我可以点击"新建工作流"按钮
- 我可以输入工作流标题（必填，≤100字符）
- 我可以输入工作流描述（可选，≤500字符）
- 我可以选择工作流分类标签
- 创建成功后，我进入工作流编辑页面

#### US-002：编辑工作流基本信息
**作为** 一位流程设计者，
**我希望** 能够修改工作流的标题和描述，
**以便** 保持工作流信息的准确性。

**验收标准：**
- 我可以在工作流列表点击"编辑"按钮
- 我可以修改标题、描述、标签
- 修改后保存，系统记录版本历史

#### US-003：删除工作流
**作为** 一位工作流创建者，
**我希望** 能够删除不再需要的工作流，
**以便** 清理工作空间。

**验收标准：**
- 我可以在工作流列表点击"删除"按钮
- 系统弹出确认对话框，提示删除后果
- 确认后，工作流被删除（或移入回收站）
- 删除操作记录在历史日志中

#### US-004：查看工作流列表
**作为** 一位普通用户，
**我希望** 能够查看我创建的所有工作流，
**以便** 快速找到需要的工作流。

**验收标准：**
- 我可以看到所有工作流的列表
- 列表显示标题、创建时间、最后修改时间、节点数量
- 我可以按创建时间、修改时间排序
- 我可以按标签筛选
- 我可以搜索工作流标题

## 7. 功能需求详情

### 7.1 节点管理模块

#### 7.1.1 功能点：添加节点

| 属性 | 描述 |
|------|------|
| **用户故事** | US-017, US-018, US-019, US-020, US-021, US-022 |
| **前置条件** | 用户已登录，处于工作流编辑页 |
| **输入** | 节点类型、节点位置、节点数据 |
| **正常流程** | 1. 用户点击"添加节点"按钮<br>2. 系统弹出节点类型选择器<br>3. 用户选择节点类型（标准/判断/并行/循环/子流程/备注）<br>4. 系统根据类型展开对应的节点编辑表单<br>5. 用户填写节点信息<br>6. 点击"保存节点"，系统保存节点 |
| **节点类型选择器** | - 标准节点：通用流程步骤<br>- 判断节点：条件分支<br>- 并行节点：并发任务<br>- 循环节点：迭代任务<br>- 子流程节点：嵌套流程<br>- 备注节点：说明提示 |
| **异常流程** | - 未选择类型：提示"请选择节点类型"<br>- 必填字段为空：提示"请填写必填字段"<br>- 保存失败：提示"保存失败，请检查网络后重试" |
| **后置条件** | 节点添加成功，工作流版本更新 |
| **验收标准** | **Given** 用户在工作流编辑页<br>**When** 选择节点类型并填写信息<br>**Then** 节点成功添加到工作流中 |
| **界面元素** | - 添加节点按钮<br>- 节点类型选择器（图标+说明卡片）<br>- 动态节点编辑表单（根据类型变化）<br>- 保存/取消按钮 |

#### 7.1.2 功能点：编辑节点

| 属性 | 描述 |
|------|------|
| **前置条件** | 用户已登录，为工作流创建者，节点存在 |
| **输入** | 节点所有字段 |
| **正常流程** | 1. 用户点击节点的"编辑"按钮<br>2. 节点进入编辑模式，展开对应类型的编辑表单<br>3. 用户修改字段内容<br>4. 点击"保存"，系统更新节点并记录版本 |
| **异常流程** | - 必填字段为空：提示"请填写必填字段" |
| **后置条件** | 节点信息更新，生成新版本 |
| **验收标准** | **Given** 用户是工作流创建者<br>**When** 修改并保存节点<br>**Then** 节点更新成功并生成新版本 |

#### 7.1.3 功能点：删除节点

| 属性 | 描述 |
|------|------|

## 9. 非功能需求

### 9.1 性能需求

| 需求项 | 指标 | 说明 |
|--------|------|------|
| 页面加载时间 | ≤ 2秒 | 工作流编辑页和查看页首屏加载 |
| 节点操作响应 | ≤ 200ms | 添加、编辑、删除节点的操作反馈 |
| 连接拖拽响应 | ≤ 100ms | 节点连接拖拽的实时反馈 |
| 并发用户数 | ≥ 1000 | 系统支持同时在线用户数 |
| 导出响应时间 | ≤ 5秒 | Markdown导出完成时间 |
| 历史版本加载 | ≤ 1秒 | 版本列表加载时间 |

### 9.2 安全需求

| 需求项 | 要求 |
|--------|------|
| 数据加密 | 工作流数据存储加密（AES-256） |
| 传输安全 | 全站HTTPS，TLS 1.3 |
| 访问控制 | 严格的权限校验，防止越权访问 |
| 分享安全 | 分享链接使用随机Token，支持有效期设置 |
| 输入校验 | 所有用户输入进行XSS和SQL注入防护 |
| 审计日志 | 记录关键操作（创建、修改、删除、分享） |

### 9.3 兼容性需求

| 需求项 | 要求 |
|--------|------|
| 浏览器支持 | Chrome 90+、Firefox 88+、Safari 14+、Edge 90+ |
| 移动端适配 | 支持iOS Safari、Android Chrome的查看模式 |
| 响应式设计 | 编辑页支持1280px以上，查看页支持768px以上 |
| 导出兼容 | Markdown兼容GitHub Flavored Markdown |

### 9.4 可用性需求

| 需求项 | 要求 |
|--------|------|
| 系统可用性 | ≥ 99.9% |
| 数据备份 | 每日自动备份，保留30天 |
| 故障恢复 | RTO ≤ 30分钟，RPO ≤ 5分钟 |
| 降级策略 | 核心功能（查看、导出）优先保障 |
